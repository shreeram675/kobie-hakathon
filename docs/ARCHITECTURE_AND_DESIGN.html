1Project Overview

Kobie is an end-to-end autonomous intelligence agent that researches and synthesises competitive profiles for consumer loyalty programs. Given a program name (or a fuzzy description), Kobie resolves the identity, plans a targeted search campaign, retrieves and scrapes evidence from multiple source types, extracts schema-grounded facts, adjudicates conflicts across sources, and produces a narrative brief plus an interactive grounded Q&A interface — all without relying on LLM training memory for factual claims.

1
Input Resolution
LLM-backed identity validator resolves ambiguous program names to a canonical identity, with up to 3 clarification rounds.
2
Research Planning
Up to 15 targeted queries planned by Gemini across 10 source types with coverage and priority metadata.
3
Evidence Extraction
Schema-grounded facts with per-field confidence scores and source attribution for every extracted value.
4
Conflict Adjudication
5-step strategy ladder: deterministic resolution to adversarial debate with judge verdict and human-review queue.
5
Narrative Synthesis
600–900 word analyst brief synthesised from adjudicated evidence; never from LLM training memory.
6
Grounded Q&A
Conversational interface with answers strictly traceable to field_report extracted evidence.
Implementation: FastAPI backend (server.py)  ·  LangGraph pipeline (pipeline/graph.py)  ·  SQLite persistence (core/db.py)  ·  Next.js 14 frontend (frontend/)
2Problem Statement

Competitive intelligence for loyalty programs is uniquely difficult. Unlike most industries, loyalty programs are characterised by:

Fragmented
No single authoritative source covers all mechanics. Earn rates, tier thresholds, partner lists, and valuations are distributed across official T&Cs, blogs, forums, and third-party comparison sites.
Volatile
Earn rates, tiers, and valuations change frequently. High-volatility fields (HIGH_VOLATILITY_FIELDS) require recency-aware adjudication and explicit staleness warnings.
Contradictory
Official program pages, independent blogs, and user forums routinely disagree on the same fact — requiring explicit conflict detection and adjudication rather than simple majority rules.
Expensive
Manual analysis requires analysts to cross-reference many pages, assess authority, reconcile contradictions, and produce a brief — a process that can take hours per program.
Key constraint: "Never use LLM training memory as a source of loyalty-program facts." Every claim must be traceable to a retrieved, scraped source URL.

1
Resolve ambiguous input to canonical identity (max 3 clarification rounds, 5-min timeout each)
2
Plan and execute up to 15 targeted queries per program across 10 distinct source types
3
Extract schema-conformant facts with EXTRACTED / NOT_FOUND / AMBIGUOUS status and confidence per field
4
Resolve conflicts via cheapest-sufficient strategy ladder from deterministic merge to adversarial debate
5

Two-program comparison with category-level verdicts and a comparative brief
7
Track per-stage cost (tokens/USD), cache completed runs, survive server restarts via SQLite WAL
 
4High-Level Architecture

Kobie is structured as five horizontal layers. The frontend communicates with the FastAPI backend via REST polling; the backend spawns per-run background threads that execute the LangGraph pipeline; the pipeline reads and writes to SQLite and calls external services.

5End-to-End Data Flow
1
Submission
POST /api/run creates a RunRecord, assigns a run_id, and starts a background thread.
2
Validation
input_validator_node sends raw input to Gemini (validation.py). Returns resolved / needs_clarification / rejected. Max 3 clarification rounds; 5-minute timeout per round via threading.Event.wait.
3
Cache Check
core/db.py::find_program_snapshot looks up a prior completed run by normalized program name. Cache hits skip to narration with the stored state.
4
Query Generation
query_generator_node calls Gemini and produces up to 15 SearchQuery objects, each annotated with source_type and field_query_map.
5
App Ratings Prefetch
app_ratings_node calls Google Play Scraper and iTunes Search API directly — no Tavily budget consumed.
6
Retrieval
retrieval_node fans all queries to Tavily (5 results/query), canonicalises URLs, and deduplicates by canonical URL to eliminate redundant scraping.
7
Web Enrichment
web_enrichment_node seeds direct URLs (official T&Cs, Trustpilot, app store pages) plus a Wikipedia summary block — zero Tavily budget spent.
8
Firecrawl Scraping
firecrawl_node selects top 20 URLs via round-robin across source types (select_urls_for_firecrawl) and scrapes each to clean markdown, handling JS-rendered pages and PDFs.
9
Ingestion
ingest_node: raw documents written to raw_store → chunker.py (600-word target, 1 500-word hard cap) → extractor.py (two-phase Gemini extraction) → normalizer.py → assembled FieldReport.
10
Adjudication
adjudication_node detects per-field conflicts and runs the 5-step strategy ladder described in Section 9.2.
11
Narration
narrator_node calls Gemini with the adjudicated FieldReport and produces the 600–900 word analyst brief.
12
Persistence & Caching
Serialised state saved to run_snapshots (keyed by program_name_normalized) and to runs (keyed by run_id) for history reconstruction.
13
Conversation
Frontend polls GET /api/run/{run_id}. Users submit questions via POST /converse; converse.py answers using final_brief + field_report through Groq.
14
Comparison Mode
Each program runs the full pipeline independently in sequence. generate_comparison_brief + compare_claim_sets produce category-level verdicts and a comparative brief.

12Design Decisions & Trade-offs
DecisionRationaleTrade-off
Thread-per-run (no Celery/RQ) Eliminates broker dependency, simplifies dev/ops for single-node deployment STORE lives in memory; horizontal scaling requires externalising state to Redis
LangGraph StateGraph Typed AgentState, explicit conditional routing, and composable node functions make the pipeline auditable Heavier abstraction than a simple function chain; adds LangGraph dependency
SQLite WAL (no Postgres) Zero-ops embedded persistence; WAL mode handles concurrent API + pipeline thread reads/writes safely Single-file; not suitable for multi-node; no full-text search index
Gemini for extraction, Groq for debate Gemini's JSON mode and low cost/token make it ideal for high-volume structured extraction; Groq's low latency suits multi-call debate loops Two provider dependencies; API key pool management adds code complexity
Cheapest-sufficient adjudication ladder Avoids LLM calls for conflicts resolvable by deterministic rules (identical values, confidence gap, field-type strategy) Ladder logic is bespoke; adding new field types requires updating strategy dispatch
Heuristic chunk selection (no embeddings) Avoids embedding API dependency and vector DB; field_query_map from query planning guides chunk relevance scoring ChunkRef.embedding_hash is reserved; switch to semantic retrieval requires migration effort
Polling over WebSocket/SSE Simpler server implementation; TanStack Query handles retry/backoff; no persistent connection state Higher request volume; latency of status updates bounded by poll interval
Snapshot cache by normalised name Repeated queries for the same program reuse the completed run, saving API cost Cache must be explicitly invalidated when fresh data is needed; staleness not auto-detected
No auth layer on API Simplifies single-user local deployment Not deployable in multi-user or public contexts without adding authentication/authorisation middleware
13Scalability, Reliability, Security & Observability
Scalability
  • Single-process / single-node; STORE is an in-memory dict guarded by STORE_LOCK
  • Thread-per-run model; compare-mode runs programs sequentially
  • No external broker or queue in current implementation
  • Horizontal scaling path: externalise STORE to Redis, move run execution to Celery/RQ
Reliability
  • checkpoint() runs PRAGMA wal_checkpoint(TRUNCATE) on FastAPI shutdown
  • server.sh integrity check + db_corrupted_backup/ on corrupt DB detection
  • Per-program result caching via run_snapshots survives restarts
  • All nodes catch exceptions → PipelineError(stage, message) → UI error state
  • Clarification and cache waits bounded to 300 s via threading.Event.wait
Security
  • API keys loaded from .env via python-dotenv; reject_placeholders=True rejects "your_..." values at startup
  • CORS restricted to localhost:3000/3001 and 127.0.0.1 equivalents
  • No authentication layer (gap before non-local deployment)
  • Extraction and debate prompts include "hallucination fence" grounding instructions
Observability
  • CostLedger: per-provider/stage call counts + token usage → USD via pricing constants
  • UI_STAGES tracked per run, surfaced to frontend via polling response
  • logs/backend.log, logs/frontend.log, logs/debate_debug.log (FileHandler in debate_engine.py)
  • Per-stage cost breakdown visible in the frontend cost panel
14Error Handling & Retry Strategy
  • Node-level isolation — every graph node catches exceptions internally and writes a PipelineError delta. The background thread continues without crashing; the UI reflects an error state for that stage.
  • Firecrawl partial failure — the scraper continues if individual URL fetches fail; the node only raises if zero URLs succeed overall.
  • Groq rate limiting — debate_engine.py::call_groq retries up to pool_size × 2 attempts, rotates keys on HTTP 429, and permanently evicts keys on HTTP 401 via _remove_client_from_pool.
  • Judge output parsing — parse_judge_output tolerates malformed or partial JSON from the debate judge, falling back to FLAG_FALLBACK_REASONING to ensure the conflict always has a recorded outcome.
  • Non-fatal enrichment failures — web_enrichment_node treats individual URL-seeding and Wikipedia fetch errors as non-fatal, logging warnings and continuing with available evidence.
  • User-triggered cancellation — record.stop_event is checked before every stage transition. In-flight stages are marked as errors and run_status is set to cancelled.
  • Debate engine fallback — if the debate engine raises an unhandled exception, the conflict is converted to a FLAG with deciding_factor: "unresolvable" and confidence 0.40 rather than failing the run.
15Future Enhancements
1
Embedding-based Retrieval
ChunkRef.embedding_hash and the chunks table schema are already forward-compatible. Moving from heuristic scoring to semantic vector search would improve extraction recall on long, multi-topic pages.
2
Multi-instance Scaling
Externalise STORE to Redis and move run execution to Celery or RQ to support parallel runs across multiple server instances with shared state.
3
Authentication & Authorisation
The API has no auth layer. Required before any non-local deployment. Candidate approach: OAuth2/OIDC middleware on FastAPI with per-user run isolation.
4
Streaming Updates via WebSocket / SSE
Replace the current polling model (GET /api/run/{id}) with a WebSocket or Server-Sent Events channel for lower-latency stage-progress updates and reduced request volume.
5
Crash-resumable Runs
Persist AgentState to the database after each completed node (not only on final completion) to enable mid-run restart recovery after a server crash.
6
verification.py Consolidation
The standalone confidence-scoring module is a candidate for merge with conflict_adjudicator.py to reduce the number of conceptually overlapping scoring paths.
7
extraction.py Retirement
The legacy extraction.py is superseded by extractor.py as the active runtime path. The older module should be formally deprecated or removed to avoid confusion about which path is canonical.
16Conclusion

Kobie demonstrates that a rigorous, source-grounded intelligence pipeline can be built from commodity LLM APIs and open-source tooling without a vector database or bespoke model. Its design is governed by a single architectural constraint: "Never hallucinate a fact."

Every extracted value carries a source_url. Conflicts between sources are resolved through an auditable ladder of deterministic strategies — using the cheapest sufficient method — before falling back to metadata-only adversarial debate. The narrative brief and the grounded Q&A interface are both constrained to adjudicated evidence only; the LLM is used as a synthesiser and writer, not as a knowledge store.

The result is an agent whose outputs can be independently verified: every claim traces back to a retrieved page, and every conflict resolution decision is recorded with its strategy, judge reasoning, and confidence score. This auditability is the architectural property that distinguishes Kobie from a simple chatbot wrapper over a search API.
