Problem & Our Approach

💡 The Core Challenge

  • Loyalty programs are documented inconsistently across official pages, T&C PDFs, review forums, and SEO blogs
  • Manually researching one program to a competitive standard takes hours and goes stale quickly
  • Sources contradict each other — earn rates, point values, and tier rules differ site-to-site
  • Programs rebrand and revise rules; no existing tool tracks temporal drift automatically

⚙️ Our Key Insight

  • Treat every extracted fact as a claim with provenance, not a value to overwrite
  • Use an adversarial debate engine (Advocate A vs B → LLaMA Judge) to resolve conflicts transparently — the LLM can never invent a third value
  • Separate retrieval, scraping, and extraction so each stage can be swapped independently
  • Rejected hallucination at the root: null when unknown, never guessed

DATA QUALITY: Web coverage for smaller programs (regional airlines, fuel retailers) is sparse; sentiment fields from Reddit/Trustpilot can be noisy. We addressed this by tiering query budgets — spending retrieval effort on genuinely hard fields (earn rates, point values, differentiators) and flagging unsupported fields as null / manual_review_needed rather than fabricating values. Source-type round-robin ensures the URL budget spans official, financial, forum, and review sources rather than collapsing into an SEO echo chamber.

Business Impact & Activation

🎯 Who This Is Built For

  • Loyalty program strategists — benchmark earn/burn structures across 10+ competitors in minutes, not days
  • Kobie consultants — enter a client meeting pre-loaded with a grounded, sourced competitive brief for any program worldwide
  • Product & partnership teams — identify partnership gaps and differentiator weaknesses at the field level
  • Analysts — trace every extracted claim to a source URL and access date; no black-box outputs

📈 The Business Case

  • Manual competitive research: 4–8 hours per program. ACI Agent: <5 minutes
  • Provenance on every claim means compliance-safe use in client deliverables
  • Adjudication audit trail (advocate arguments + judge verdict) makes the reasoning reviewable — not a black box
  • Temporal ordering via retrieved_at means stale data is flagged, not silently overwritten
  • Schema-driven design: add a new field to the schema config → extraction covers it on the next run without rewriting pipeline logic

Live Demo Highlights

Show, don't tell. Walk the judges through a live run or key screenshots.

1. End-to-End Pipeline Execution

Type any loyalty program name (e.g. "Marriott Bonvoy") into the Input Verifier tab → the 6-node LangGraph pipeline fires automatically. The Pipeline Inspector tab shows live node-by-node status, query list, scraped URL count, chunk count, and extracted field coverage — all in one view.

2. Conflict Adjudication in Action

Open the Pipeline Inspector after a run. Expand a HIGH-volatility field (e.g. earn_mechanics.base_earn_rate). Show two contradicting claims from independent sources. Demonstrate auto-resolve (gap > 0.20) vs. a live 5-step adversarial debate — Advocate A, Advocate B, Rebuttal A, Rebuttal B, then the LLaMA judge returning a structured JSON verdict.

3. Compare Mode — Two Programs Side by Side

Switch to the Compare tab. Enter two programs (e.g. "Delta SkyMiles" and "United MileagePlus"). The pipeline runs both in parallel via ThreadPoolExecutor and renders a structured side-by-side field comparison. Point out where the adjudicated values differ and which source won each conflict.

4. Converse Tab — Ask Field-Level Questions

Switch to the Converse tab. Ask: "What is the point expiry policy?" or "Which partners are listed?". Show answers grounded in stored claim evidence — not LLM memory — with source URL attribution.

Learnings, Limitations & Next Steps

🎓 What We Learned

  • Source contradiction is the norm, not the exception — earn rates and point values differ on almost every program across 3+ sources
  • The biggest quality lever was volatility-weighted confidence, not the extraction LLM itself — scoring recency higher for earn rates eliminated most false auto-resolves
  • Heading-based chunking dramatically outperformed fixed-window chunking for loyalty T&C pages — structure matters more than size
  • Query planning with explicit field-difficulty tiers reduced Firecrawl spend by ~30% vs. generic queries while improving coverage on hard fields

⚠️ Honest Limitations

  • Sentiment fields are sparse for smaller or regional programs — forum coverage is thin
  • No Temporal Ledger: retrieved_at preserves recency ordering but there is no structured audit trail of value changes over time
  • Brief generation is scaffolded — narrator not yet wired to the adjudicated field report
  • Confidence calibration for the debate engine is empirically tuned on a small program set; high-conflict programs with thin evidence may still misfire
  • SEO echo chambers can survive deduplication if the same claim propagates across many sites — lineage-based dedup not yet implemented

🚀 If We Had More Time…

  • Wire brief generation to the adjudicated field report — emit a structured analyst brief with entailment check per sentence
  • Temporal Ledger — append-only SQLite table tracking field value changes with date stamps, enabling "what changed since last month?" queries
  • Full Converse module — upgrade from keyword-match QA to Groq-powered RAG over the canonical field report
  • What-if simulation — "if Program X lowered their earn rate by 0.5x, how would it compare to competitors?" — modelled over adjudicated values
